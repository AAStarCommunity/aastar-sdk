> **SDK fix this surface forced (real v0.20.0 encoding break, same discipline as the InitConfig
> fix).** The first submission reverted `AA24 signature error`. Root cause:
> `AAStarAirAccountBase._validateSignature` (v0.20.0) routes on `signature[0]` as an **algId prefix
> BEFORE** the raw-65-byte fallback (`AAStarAirAccountBase.sol:596-647`). A raw 65-byte ECDSA sig
> whose first byte happens to equal an algId const — e.g. `r`'s high byte `0x02` == `ALG_ECDSA` —
> is misrouted into the explicit-ECDSA branch, which REQUIRES 66 bytes, so it returns
> `SIG_VALIDATION_FAILED`. Fix: emit the deterministic **algId-prefixed** signature
> `[0x02][r||s||v]` (66 bytes) — the same format the SDK's hardware signers already use
> (`packages/airaccount/src/auth/hardware/ledger.ts`). Applied at the `beta1` signing site.
> A second submission then reverted `AA33 Paymaster__InsufficientBalance` because the test's
> 200-token deposit was below the validation-time `requiredTokenAmount` (≈292 at ~8.5 gwei /
> ETH≈$1730 / token=$0.02); the deposit was raised to 1000.

## Surface: KMS create→sign (TEE) — ⚠️ BLOCKED (infra, not encoding)
Run: `KMS_E2E=1 … kms-account-e2e`. The SDK `KmsManager` wrapper reached the live endpoint
`https://kms.aastar.io` and was rejected at authentication:

```
❌ KMS account E2E FAILED: Request failed with status code 401
   server: {"error":"Missing API key"}
```

Root cause: the live KMS backend requires an API key (`KMS_E2E_API_KEY`) that is **not provisioned**
in `.env.sepolia`. This is a credentials/infra block, NOT a v0.20.0 encoding problem — the SDK
client built and sent the request correctly. To unblock: provision `KMS_E2E_API_KEY` for
`kms.aastar.io` and re-run.
