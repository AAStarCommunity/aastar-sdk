# v0.36.1 — Tier-1 single-ECDSA algId 0x02 framing, LIVE on-chain acceptance (#273)

**The #273 SDK fix proven on-chain against the LIVE deployed account, on Sepolia.**

airaccount-contract v0.25.0 removes the raw-65 ECDSA fallback, so the SDK's
`generateTieredSignature(tier=1)` now frames Tier-1 as `[algId 0x02][r][s][v]` = 66 bytes
(PR #275). The deployed stack the SDK talks to is **v0.24.0** (the raw-65 fallback is still present
there), so the real risk a signing-path change carries is: **does the currently-deployed v0.24.0
account ACCEPT the new 0x02-framed Tier-1 signature** — i.e. does shipping 0.36.1 keep Tier-1 working
today, and not break it during the v0.24 → v0.25 transition?

Software-reproducible — NO DVT / bundler / device passkey (Tier-1 is a pure owner ECDSA over
userOpHash). The signature is produced by the REAL SDK path under test:
`BLSSignatureService.generateTieredSignature({ tier: 1 })` backed by a `LocalWalletSigner`, which
internally calls the shared `packEcdsaAlgId` framer (#273).

Script: `tests/regression/onchain-evidence/tier1-ecdsa-algid-e2e.ts`

## What this proves
1. **The SDK emits the correct frame**: `generateTieredSignature(tier=1)` returns `[0x02]‖r‖s‖v` = 66 bytes
   (asserted in-script before submission), via the shared `packEcdsaAlgId` helper (#273).
2. **The live v0.24.0 account ACCEPTS it** (`validateUserOp == 0`) — so shipping SDK 0.36.1 does **not**
   break Tier-1 on today's accounts, and is forward-ready for the v0.25.0 fallback removal.
3. **The check is real, not a rubber-stamp**: a 0x02-framed signature from the wrong signer is **rejected**
   (`validateUserOp == 1`), and the old raw-65 output is shown still-accepted on v0.24.0 (isolating the
   change as forward-compat, not a break-today).

## Coverage note (honest gaps)
- The **raw-65 REJECTION** that v0.25.0 introduces cannot be proven on-chain yet — no v0.25.0 account is
  deployed on Sepolia (the SDK is pinned to v0.24.0). Row 2 above records that v0.24.0 still accepts raw-65;
  the SDK change is what makes Tier-1 keep working once a v0.25.0 account exists. Re-run this probe against a
  v0.25.0 account when airaccount deploys one, and assert row 2 flips to `!= 0`.
- The **Tier-2/3 composite regression** (`tier3-composite-e2e.ts`, §4-mandated for any tiered-path change)

  and the 856-test unit suite covers those layouts. Re-green `tier3-composite-e2e.ts` once the DVT sign
  endpoint is reachable again.
