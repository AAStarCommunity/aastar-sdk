# v0.32.1 — guard-read regression fix for v0.22.0 accounts (#254)

## Bug (regression, production-bricking)
`readAccountGuardAddress` (used by guard-checker, which `prepareTransfer` runs on EVERY transfer + config
op) read `account.getConfigDescription().guardAddress`. **v0.22.0 accounts REMOVED `getConfigDescription()`**
(only ForceExitModule exposes one, with different semantics) → the call reverts → `prepareTransfer` reverts
→ ALL transfers / config (persona apply + real transfers) on v0.22.0 accounts were blocked.

## Fix
Read `guard()` directly (same value the struct's `guardAddress` field carried) — already used by
guard-state-reader. One-line change in `packages/airaccount/src/server/providers/typed-reads.ts`.

## On-chain proof (Sepolia, v0.22.0 account `0x7c458906…`)
- `guard()` → `0x185923fda8728e2b1CF781e78c2Da995A54A8BC0` (non-zero) ✅
- `getConfigDescription()` → **REVERTS** ("function getConfigDescription reverted") ✅ — confirms the regression

## Tests
`typed-reads.guard.test.ts` (2): readAccountGuardAddress reads `guard()` and NEVER calls
getConfigDescription; propagates a zero guard. Full suite 0 failed (airaccount 833 / core 447); build green.
