# On-chain evidence — v0.31.0: createAccountWithPasskey (KMS relay-mode passkey-at-birth, #249)

New SDK feature (no contract change). Closes the gap 0.30.0 left: v0.22.0 passkey-at-birth was only
proven for EOA owners (DIRECT mode). KMS accounts hold the owner key in a TEE → must use RELAY mode
(owner EIP-191-signs the CREATE_ACCOUNT digest; a deployer relays + pays gas; msg.sender != owner).

Network: Sepolia (11155111). Factory `0x0eb0E7a61d5D9e03bc3578f8C1b0d9f40cc0a5B9` (v0.22.0).

## Acceptance (FEATURE-MET)

| Scenario | Path | Expected | Result | Evidence |
|---|---|---|---|---|
| KMS-style relay passkey-at-birth deploy | owner (fresh key, stands in for KMS) signs `buildCreateAccountHash` EIP-191 → deployer relays `createAccount` (8-arg, ownerSig) | account deploys at the predicted addr; `p256KeyX()==ownerP256X`; `validator()!=0`; `createNonces(owner) 0→1` | ✅ **FEATURE-MET** | account `0xC51DBbBb08Dcf57c0765a53F59d6Ddef0ab7B826`; relay tx `0xf6d5f3474d59939a710def0386e62d4aa0fb255214793766560d090173db04ac` (status success); p256KeyX==passkey; validator `0xfcDfd17a…`; nonce 0→1 |

**Why this proves byte-exactness:** the v0.22.0 factory recovers the relay `ownerSig` against
`keccak256(abi.encode("CREATE_ACCOUNT", chainId, factory, owner, salt, ownerP256X, ownerP256Y,
_getConfigHash(config), nonce, deadline)).toEthSignedMessageHash()`. If `buildCreateAccountHash`
mis-encoded ANY field (order/type) or `configHashFromInitConfig` diverged from the contract's internal
`_getConfigHash`, `ecrecover != owner` → the deploy reverts `InvalidOwnerSignature`. A SUCCESSFUL relay
deploy is therefore proof the SDK replica matches the on-chain hash.

Runner: `tests/regression/onchain-evidence/createaccount-relay-passkey-e2e.ts` (software owner stands in
for the KMS signer — same EIP-191 personal-sign; deployer = `PRIVATE_KEY_JASON`).

## Adversarial review (Codex §5)
- CHECK 1 (digest byte-exactness vs contract `_getConfigHash` + CREATE_ACCOUNT preimage): **CLEAN**.
- CHECK 2 **HIGH (fixed)**: the digest was passed to `signer.signMessage` as a `0x…` STRING — the KMS
  adapter's `hashMessage` treats a string as UTF-8, so KMS would EIP-191-sign the ASCII text, not the raw
  32-byte digest → relay reverts `InvalidOwnerSignature`. Fixed: pass `hexToBytes(hash)` (matches the
  bls-signature-service convention; LocalWallet `{raw}` already handled it, which masked it in the e2e).
- CHECK 3 **MEDIUM (fixed)**: no guardian validation → a dup/owner guardian deploy-reverts on-chain
  (`DuplicateGuardian`) while `getAddress` does not, stranding a pre-funded address. Fixed: fail fast on
  `guardian == owner` + duplicate ECDSA guardians before predict/deploy (+2 unit tests).
