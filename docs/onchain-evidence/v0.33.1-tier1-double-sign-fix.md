# v0.33.1 — Tier-1 double-sign fix (#259)

## Bug (0.33.0 regression, blocks arming / Tier-1 owner-signed ops)
On a KMS `challenge_mode: strict` signer, a WebAuthn ceremony assertion is SINGLE-USE (correct anti-replay).
`applySignature` built the #257 DVT `ownerAuth` for ALL tiers — but Tier-1 is a raw owner ECDSA over
`userOpHash` that `generateTieredSignature` signs directly and uses NO DVT/BLS. So the assertion was fed to
`signMessage` TWICE (ownerAuth + the Tier-1 op signature, same owner + same userOpHash):
```
KMS /SignHash → 200  (challenge consumed)
KMS /SignHash → 400  "Challenge not found or expired"  (same assertion, spent challenge)
```
KMS is correct; the bug was the SDK signing one ceremony twice.

## Fix
Build a `dvtRequest` (and thus `ownerAuth`) ONLY for Tier-2/3 (the tiers that talk to the DVT). For those,
`ownerAuth` is the sole ceremony sign — the composite uses the device passkey + BLS + guardian and
`generateBLSSignature` skips `aaSignature`/`messagePointSignature` under `skipOwnerOpSignature` (#258 M1).
Tier-1 now signs exactly once. General principle: one ceremony assertion → exactly one `signMessage`.

## Verification
- Unit (`transfer-manager.test.ts`): Tier-1 builds NO dvtRequest (no second sign); Tier-2/3 = exactly one
  ceremony sign (ownerAuth). airaccount 839 / core 447 pass; build green.
- The KMS strict-challenge 400 is a live-KMS behavior (a LocalWalletSigner e2e has no single-use challenge),
  so acceptance is the unit regression + the root-cause + the B3 re-run on the real KMS/board.
