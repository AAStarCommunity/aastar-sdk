## #257 — DVT compatibility (sign format + node discovery) + P2P seam
The redeployed DVT (v1.7) rejects the legacy `{ message }` sign body and validates owner authorization
before co-signing; discovery returned 1 localhost node (Tier-3 needs ≥2).
- **Discovery** (`getAvailableNodes`): iterate all seeds, use the EXTERNAL seed URL as the apiEndpoint
  (the peer's registered apiEndpoint is `localhost:400x`), dedupe by nodeId → ≥2 external nodes.
- **Sign format**: POST `{ userOp (packed RPC), ownerAuth }`. `ownerAuth` = the owner's EIP-191 sig over
  `userOpHash` — a DVT authorization credential, produced in the SUBMIT flow (`buildDvtRequest`) and
  threaded as a transport param; it is NOT part of the on-chain composite. The composite-signature
  functions stay pure (P256 + BLS + guardian).
- **P2P-migration seam**: the DVT transport is isolated in ONE method, `_coordinateBlsAggregate`. A future
  self-organizing P2P transport swaps just it — no change to the composite assembly, packers, or contract.

**On-chain proof (live dvt1/2/3.aastar.io)**:
```

                        SDK { userOp, ownerAuth } body → all 3 co-sign → aggregate 256B (owner=JASON).
dvt-realnode-e2e:       same format → aggregate → EntryPoint validate() == 0 (ACCEPTED on-chain).
```
