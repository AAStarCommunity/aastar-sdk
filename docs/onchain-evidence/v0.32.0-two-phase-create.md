# v0.32.0 — two-phase prepare/submitCreateAccountWithPasskey (separated-ceremony KMS, #249)

New SDK feature (no contract change). Closes the KMS gap in 0.31.0's `createAccountWithPasskey`: the
single method works for EOA owners (inline sign) but DEADLOCKS for KMS owners — the WebAuthn ceremony's
challenge must commit to the CREATE_ACCOUNT digest (`beginCeremony` computes
`challenge = SHA-256(nonce ‖ sign-digest(message))`), but the digest depends on nonce/deadline resolved
INSIDE the method, so the caller can't pre-make the assertion. Same reason transfers are two-phase.

## API (`@aastar/sdk/kms` → `AccountManager`)
- `prepareCreateAccountWithPasskey(userId, params)` → `{ createId, predictedAddress, challenge (digest),
  challengeId?, publicKeyOptions?, nonce, deadline, alreadyDeployed }`. For KMS signers begins the ceremony
  over the digest; the frontend runs `navigator.credentials.get(publicKeyOptions)`.
- `submitPreparedCreateAccount(createId, { deployerWallet, signerCtx })` → signs the prepared digest with
  the ceremony assertion (KMS owner key) + relays `createAccount` → deployed `AccountRecord`.

## Coverage
- **Unit (12, `account-manager.passkey.test.ts`)**: inline (EOA) 6 + two-phase 6 — prepare returns the
  digest as challenge; submit signs the prepared digest as RAW BYTES + relays + single-use; KMS
  `beginCeremony` commits to the digest (identical bytes prepare↔submit); nonce-advanced abort; single-use
  on relay failure; unknown/expired id.
- **On-chain (Sepolia v0.22.0)**: the digest + relay deploy mechanism is proven by
  `createaccount-relay-passkey-e2e.ts` (account `0xC51DBbBb…`, relay tx `0xf6d5f34…`, p256KeyX==passkey,
  nonce 0→1). The two-phase split only changes WHEN the (identical) digest is signed; with LocalWalletSigner
  it produces the same ownerSig as the proven inline path. The AccountManager reuses the same
  EthereumProvider methods as the existing on-chain-tested create flows (createAccountWithP256Guardians).
