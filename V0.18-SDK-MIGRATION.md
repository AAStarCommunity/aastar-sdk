# AirAccount v0.18 — SDK migration

AirAccount contracts were redeployed to Sepolia on **2026-06-14** (full stack — the account is
non-upgradable, so every contract is new). All work-streams WS-A..G merged to `main`, plus the
**#45 CRITICAL** BLS↔userOpHash binding and the **#82** factory EIP-3860 initcode fix.

This branch (`feat/v0.18-contracts`) does the **safe mechanical** part of the sync:

- refreshes the vendored AirAccount ABIs in `packages/core/src/abis/*` from the contract repo
  (`out/<Contract>.sol/<Contract>.json` per contract; merged `abi/AAStarAirAccountV7.full.json`
  for the account), and
- updates the Sepolia AirAccount addresses in `packages/core/src/addresses.ts`.

`@aastar/core` builds cleanly (`tsc`) with these ABIs. Anything that requires **behavioural** SDK
code changes (e.g. a packer for a new signature layout, or a deploy helper that constructs the
factory) is called out in "Follow-ups" below and is intentionally NOT auto-applied.

> 🔴 The WIP branch `sync/superpaymaster-v5.3.3-beta.3` is untouched. SuperPaymaster addresses in
> `addresses.ts` were left exactly as-is — this sync only changes the AirAccount contract set.

## ABI changes (done in this branch)

### `AAStarAirAccountFactoryV7` — NEW constructor (BREAKING for deploy code) — #82
```
constructor(
  address _implementation,        // NEW — arg 1; was deployed inside the ctor before
  address _entryPoint,
  address _communityGuardian,
  address[] defaultTokens,
  AAStarGlobalGuard.TokenConfig[] defaultConfigs
)
```
The implementation is now **injected as arg 1** instead of being deployed by the factory constructor
(initcode 49,134 → 13,324 bytes, under the EIP-3860 cap). Any SDK/script that *deploys* the factory
must pass a pre-deployed implementation address. New error: `ImplementationRequired`.

### `AAStarBLSAlgorithm` — #45 + Ownable2Step
- `+ aggregator() view returns (address)` — single protocol-level batch-aggregator pointer. The
  account READS this during BLS validation (non-zero ⇒ defer to batch aggregator; zero ⇒ inline
  single-op binding). There is **no per-account aggregator field** and **no account-side
  `blsAggregator()` getter anymore** — read the aggregator from `AAStarBLSAlgorithm.aggregator()`.
- `+ setAggregator(address) onlyOwner` — Safe-only on mainnet.
- `+ acceptOwnership()` / `+ pendingOwner()` — two-step ownership (Ownable2Step).
- The `validate(hash, signature)` calldata **no longer accepts a caller-supplied `messagePoint`**; the
  point is recomputed on-chain from `userOpHash` via RFC 9380 `hash_to_curve`. See "Follow-ups".

### `AAStarBLSAggregator` — NEW constructor
```
constructor(address _blsAlgorithm, address _entryPoint)
```
The aggregator now takes the canonical EntryPoint so it can re-derive each op's `userOpHash`.

### `AAStarAirAccountV7` (merged full ABI, 153 entries) — module-install timelock surface
- `+ moduleManagementNonce() view returns (uint256)` — guardian-signed install/uninstall replay nonce
  (WS-A #75). The guardian signature domain now includes this nonce + a version epoch.
- `+ proposeModuleInstall(...)` and `+ pendingModuleInstall(...)` for the optional install timelock (WS-D #58):
  ```
  pendingModuleInstall() returns (
    address module, uint8 moduleTypeId, uint40 proposedAt, uint40 executeAfter, bytes32 initDataHash
  )
  ```
- `+ ValidatorAlreadySet` error (set-once validator, #45).

### `AirAccountExtension` (51 → 79 entries)
Adds the module-install-timelock functions + errors: `ModuleInstallAuthChanged`,
`ModuleInstallTimelockTooLong`, `ModuleInstallProposalExpired`.

### Narrowed storage getters (storage packing, WS-E)
- `AAStarGlobalGuard.tokenConfigs(token)` → `(uint128 tier1Limit, uint128 tier2Limit, uint256 dailyLimit)`
  (was 3×`uint256`). NOTE: the guard ABI is **not vendored in the SDK** — flagged here for any
  consumer that decodes guard config off-chain (#82).
- `SessionKeyValidator.sessionStates(account, key)` → `(uint48 windowStart, uint32 callCount, uint32 prevCount)`
  (narrowed for the sliding-window velocity limiter, WS-C #57). `sessionStates_p256` has the same shape.

### New session-key surface (WS-C #83/#57)
`sessionKeyCount(account)`, per-account cap of 50, sliding-window velocity limiter. New error
`TooManySessionKeys`. The velocity limiter is documented as **approximate** at window boundaries.

## Errors named in the original task vs reality

The task listed an `AggregatorDisabled` error. **It does not exist** in the v0.18 contract source —
when the protocol aggregator is `address(0)` the account simply falls back to inline single-op BLS
binding (no revert). `TierLimitTooLarge` lives on `AAStarGlobalGuard` (not vendored). The other
named errors all landed in the refreshed ABIs:
`ValidatorAlreadySet` (account), `ImplementationRequired` (factory),
`ModuleInstallAuthChanged` / `ModuleInstallTimelockTooLong` / `ModuleInstallProposalExpired` (extension).

## Follow-ups (NOT done here — need behavioural changes / design review)

1. **#45 BLS signature packers** — if/when the SDK packs BLS payloads (triple `0x01`, cumulative
   `0x04`/`0x05`, weighted `0x07`), it must **stop appending `messagePoint` (256 bytes) and `mpSig`
   (65 bytes)** to every BLS payload — the contract now recomputes the point on-chain. Keep the DST
   and 32-byte `userOpHash` pre-image exactly as today. Old-format sigs are hard-rejected on a length
   check. Full spec: `airaccount-contract/docs/issue45-fix1-yaa-changes.md`. This is the
   `YetAnotherAA-Validator` / DVT side; mirror it in any SDK packer.
2. **Factory deploy helper** — any SDK code or script that deploys `AAStarAirAccountFactoryV7` must
   adopt the new ctor (impl as arg 1). Account *creation* via an already-deployed factory is unchanged.
3. **`@aastar/core` version bump + downstream packages** — `packages/core` is already at `0.18.0`;
   re-export / re-publish as needed for `sdk` / `airaccount` / `enduser` consumers.
4. **Batch BLS path is OFF on this testnet deploy** (`AAStarBLSAlgorithm.aggregator() == address(0)`),
   so SDK code should treat single-op inline binding as the default and only use the aggregator when
   `aggregator()` is non-zero.
